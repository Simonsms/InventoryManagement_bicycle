# 调拨管理模块使用说明

## 1. 模块定位

调拨管理模块用于处理门店之间的库存流转，当前分为两类单据：

- `实物调拨`
  适用于商品真实从一个门店发往另一个门店的场景。
- `账面划转`
  适用于线下已经完成搬货、系统需要补记，或需要做门店间账务修正的场景。

这两类单据共用一个“调拨管理”模块，但状态流转和库存变更时机不同。

## 2. 适用场景

### 2.1 实物调拨适用于

- 分店缺货，总店或其他门店有现货，需要真实发货补货
- 展陈样车、活动车辆需要跨门店调配
- 临时销售高峰，需要从其他门店紧急调货

### 2.2 账面划转适用于

- 线下已完成搬货，但系统忘记录入，需要补做库存调整
- 历史数据录错，需要把库存从 A 店修正到 B 店
- 开业初始化、盘点后人工确认需要做门店间账面修正

### 2.3 不建议使用本模块的情况

- 正常采购入库：请走入库流程
- 正常销售出库：请走出库流程
- 单门店盘点差异修正：请走盘点或库存调整流程

## 3. 角色与权限

### 3.1 实物调拨

- 发起：店主、门店管理员
- 审批：店主
- 发货：店主，或调出门店对应的门店管理员
- 收货：店主，或调入门店对应的门店管理员

### 3.2 账面划转

- 发起：仅店主
- 审批：店主
- 执行划转：店主

### 3.3 自审例外

默认不建议自己发起自己审批。

如果系统中只有一个店主账号，且没有第二审批人可用，系统会允许账面划转出现 `自审例外`。这类单据会被明确标记，方便后续审计和追溯。

## 4. 页面使用方式

调拨管理页面顶部先按单据类型分为：

- `实物调拨`
- `账面划转`

选择类型后，再按状态查看单据列表。

### 4.1 实物调拨状态页签

- 待审批
- 待发货
- 在途待收货
- 已完成
- 已拒绝

### 4.2 账面划转状态页签

- 待审批
- 待执行
- 已完成
- 已拒绝

## 5. 业务操作说明

### 5.1 发起实物调拨

1. 进入 `调拨管理`
2. 选择顶部类型为 `实物调拨`
3. 点击 `发起调拨`
4. 填写：
   - 调出门店
   - 调入门店
   - 原因
   - 商品明细和数量
   - 备注
5. 提交后，单据进入 `待审批`

### 5.2 审批实物调拨

1. 店主进入 `实物调拨 -> 待审批`
2. 找到目标单据
3. 点击 `通过` 或 `拒绝`

审批通过后，单据进入 `待发货`，此时库存还没有变化。

### 5.3 发货

1. 调出门店对应人员进入 `实物调拨 -> 待发货`
2. 找到目标单据
3. 点击 `发货`

发货后：

- 调出门店库存会扣减
- 系统写入 `transfer_out` 流水
- 单据进入 `在途待收货`

### 5.4 收货

1. 调入门店对应人员进入 `实物调拨 -> 在途待收货`
2. 找到目标单据
3. 点击 `确认收货`

收货后：

- 调入门店库存会增加
- 系统写入 `transfer_in` 流水
- 单据进入 `已完成`

### 5.5 发起账面划转

1. 进入 `调拨管理`
2. 选择顶部类型为 `账面划转`
3. 点击 `发起账面划转`
4. 填写：
   - 调出门店
   - 调入门店
   - 原因
   - 商品明细和数量
   - 备注
5. 提交后，单据进入 `待审批`

### 5.6 审批并执行账面划转

1. 店主进入 `账面划转 -> 待审批`
2. 审批通过后，单据进入 `待执行`
3. 在 `待执行` 中点击 `执行划转`

执行后：

- 调出门店库存减少
- 调入门店库存增加
- 系统同时写入 `transfer_out` 和 `transfer_in` 流水
- 单据进入 `已完成`

## 6. 状态说明

### 6.1 `pending`

- 单据已提交
- 尚未审批
- 不改库存

### 6.2 `approved`

- 审批通过
- 实物调拨表示“允许发货”
- 账面划转表示“等待执行”
- 不直接改库存

### 6.3 `in_transit`

- 仅用于实物调拨
- 表示已经发货、货在路上
- 调出门店库存已扣减
- 调入门店库存尚未增加

### 6.4 `completed`

- 单据流程结束
- 库存变更已经全部落账

### 6.5 `rejected`

- 审批拒绝
- 单据终止
- 不改库存

## 7. 库存与流水规则

### 7.1 实物调拨

- 创建单据：不改库存
- 审批通过：不改库存
- 发货：扣减调出门店库存，写 `transfer_out`
- 收货：增加调入门店库存，写 `transfer_in`

### 7.2 账面划转

- 创建单据：不改库存
- 审批通过：不改库存
- 执行划转：一次性调整两边库存，同时写 `transfer_out` 和 `transfer_in`

### 7.3 库存校验

系统会在两个阶段校验库存：

- 创建单据时做预校验，防止明显超出库存
- 实际执行库存变更时再做一次硬校验，避免申请后库存已被其他业务占用

## 8. 风险与注意事项

- 审批通过不代表库存已经变化
- 实物调拨只有在 `发货` 后，调出门店库存才会减少
- 实物调拨只有在 `确认收货` 后，调入门店库存才会增加
- 账面划转风险高于实物调拨，必须确保原因填写准确
- 出现 `自审例外` 标签的单据，需要重点复核
- 若发货前库存已被其他业务占用，系统会阻止继续发货或执行划转

## 9. 系统规则说明

这一节主要给开发、实施或排障人员参考。

### 9.1 后端接口

- `GET /api/v1/transfers`
- `POST /api/v1/transfers`
- `PUT /api/v1/transfers/:id/approve`
- `PUT /api/v1/transfers/:id/reject`
- `PUT /api/v1/transfers/:id/ship`
- `PUT /api/v1/transfers/:id/receive`
- `PUT /api/v1/transfers/:id/execute-adjustment`

### 9.2 单据类型

- `physical_transfer`
- `book_adjustment`

### 9.3 状态机

- 实物调拨：
  `pending -> approved -> in_transit -> completed / rejected`
- 账面划转：
  `pending -> approved -> completed / rejected`

### 9.4 关键字段

- `type`：单据类型
- `status`：当前状态
- `approvedBy / approvedAt`：审批信息
- `shippedBy / shippedAt`：发货信息
- `receivedBy / receivedAt`：收货信息
- `completedBy / completedAt`：最终执行完成信息
- `selfApprovedException`：是否单店主自审例外
- `reasonCode`：业务原因分类

### 9.5 当前已知历史遗留

项目数据库里部分表仍存在早期遗留的“驼峰列 + 下划线列”并存问题。当前调拨模块已经按真实运行结构对齐，但后续如果要继续做数据库收口，建议单独安排一次结构清理，不要在业务改造中顺手混做。
